# ADR 0001: アーキテクチャ決定記録を運用する

## ステータス

Accepted

## 背景

Jubako は UI 挙動、OS 連携、ローカル永続化が結合したデスクトップアプリです。設計判断を明示的に記録しない場合、重要な判断がコードに埋没し、後続変更時の安全な見直しが難しくなります。

## 決定

アーキテクチャレベルの判断とトレードオフを記録する正本として、`docs/adrs/` 配下で ADR を運用します。

本 ADR で定義する基本方針:

- 重要な設計判断ごとに連番 ADR（`NNNN-<decision-kebab>.md`）を作成する。
- ADR には Context / Decision / Consequences / Alternatives / References を含める。
- `Proposed` / `Accepted` / `Deprecated` / `Superseded` の状態変更時は理由を明記する。

## 影響

- チームは実装制約の背景を監査しやすくなる。
- 将来リファクタ時に前提妥当性を評価しやすくなる。
- 文書管理コストは増えるが、設計判断の曖昧さは減る。

## 検討した代替案

- 判断を PR 説明文のみに残す:
  - 長期的な検索性・一覧性が低く、運用基盤として不十分なため不採用。
- 判断をコードコメントのみに残す:
  - 横断的な設計意図は局所コメントに収まりにくいため不採用。
- 外部 Wiki のみに記録する:
  - ソースコード変更と同じ履歴で管理できないため不採用。
